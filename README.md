# Паяльня PM

SPA для управління проектами і завданнями. Дизайн у стилі Slash — темна тема, obsidian + ember gold.

**[→ Демо](https://payalnya-k9u2iw23h-yaroslav-radchuks-projects.vercel.app/#/)**

---

## Стек

Vue 3 Composition API, TypeScript (без `any`), Pinia, Vite, SCSS + BEM, vue-i18n.

Для форм — vee-validate + zod: схема описується один раз, помилки з'являються після blur і при сабміті. Drag-and-drop через vuedraggable (SortableJS). Графік статистики — Chart.js через vue-chartjs.

## Архітектура

### Структура

Два view-компоненти (`HomeView`, `ProjectView`) — підключають стори і передають дані вниз. Компоненти розбиті на три папки: `common/` — перевикористовувані UI-блоки без доменної логіки, `projects/` і `tasks/` — все що прив'язане до конкретної сутності. Стори, composables і api — окремі директорії, кожна з одним рівнем вкладеності.

### Причини поділу

`common/` містить UI-примітиви без доменної логіки — вони не знають ні про проекти, ні про завдання, тому їх можна підключити будь-де. `projects/` і `tasks/` — доменні компоненти, які працюють зі своїми сторами. Такий поділ не дає доменній логіці просочуватись у загальні блоки і навпаки.

Views (`HomeView`, `ProjectView`) — тонкий шар: підключають стори, передають дані вниз через props, не містять бізнес-логіки. Всі мутації — через actions стора.

### Composables

Логіка таблиць не живе у компонентах — сортування, фільтрація і ресайз колонок винесені в окремі generic composables, кожен незалежний від інших. Це дозволяє перевикористовувати їх у різних таблицях (проекти, завдання) без дублювання. Стан фільтрів і режим відображення (table/kanban) зберігаються в `localStorage` між сесіями.

Підтверджуючі модалки (видалення, архівування) реалізовані через Promise-based `useConfirm` — виклик виглядає як `await confirm({ ... })`, відповідь повертається після вибору користувача. Це дозволило уникнути локального стану відкриття модалки в кожному компоненті.

### API-шар

HTTP-шар побудований на axios. Всі запити йдуть через `src/api/client.ts`, де налаштований `baseURL`. Проекти і завдання мають окремі модулі (`projects.ts`, `tasks.ts`) — кожен описує типізовані функції для CRUD-операцій.

## Мок-адаптер

Бекенду немає — `axios-mock-adapter` перехоплює всі HTTP-запити ще до мережі і відповідає з симульованою затримкою (~200ms).

Дані зберігаються в `localStorage`: при першому запуску записується seed-набір проектів і завдань, далі всі create/update/delete зберігають зміни туди ж — тому дані живуть між перезавантаженнями.

Щоб підключити реальний бекенд — прибрати виклик `installMockAdapter` у `src/main.ts` і змінити `baseURL` у `src/api/client.ts`. Решта коду не потребує змін.

## Що реалізовано по ТЗ

Весь обов'язковий функціонал: таблиця проектів із сортуванням, фільтрацією і ресайзом колонок, сторінка проекту з двома режимами відображення (таблиця + канбан), drag-and-drop усередині колонок і між ними зі збереженням порядку і статусу, форма створення/редагування завдань із валідацією, повний API-шар.

Весь бонусний функціонал: toast-повідомлення після кожної дії, діаграма розподілу завдань на головній сторінці, збереження стану фільтрів між перезавантаженнями, анімації переходів між сторінками, модальними вікнами і картками канбану.

Додатково зверх ТЗ: архівування і відновлення проектів, модалки підтвердження для деструктивних дій, скелетони при завантаженні, юніт-тести (Vitest + Vue Test Utils), локалізація інтерфейсу (українська / англійська), редагування проектів і завдань.
